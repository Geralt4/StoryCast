# Fix: Player Crash on Navigation (Swift Concurrency Executor Mismatch)

## Problem
App crashes with `EXC_BREAKPOINT` when navigating to player after importing an audiobook.
Initial fix resolved complete crash but Thread 25 still hits `EXC_BREAKPOINT`.

## Root Cause
`RemoteCommandHandler.swift:105` - The `MPMediaItemArtwork` closure inherits `@MainActor` isolation but is called by MediaPlayer framework on a background queue.

**Current incomplete fix (still broken):**
```swift
if let artwork = artwork {
    let capturedArtwork = artwork  // Local capture doesn't break isolation
    nowPlayingInfo[MPMediaItemPropertyArtwork] = MPMediaItemArtwork(boundsSize: capturedArtwork.size) { _ in capturedArtwork }
}
```

The closure `{ _ in capturedArtwork }` still inherits `@MainActor` isolation from the enclosing `@MainActor` method. When MediaPlayer framework calls this closure on a background queue, Swift 6 detects the executor mismatch and triggers `EXC_BREAKPOINT`.

## Solution
Mark the closure explicitly as `@Sendable` to break `@MainActor` isolation.

### Fix in `RemoteCommandHandler.swift` (line 105)

**Before (current incomplete fix):**
```swift
if let artwork = artwork {
    let capturedArtwork = artwork
    nowPlayingInfo[MPMediaItemPropertyArtwork] = MPMediaItemArtwork(boundsSize: capturedArtwork.size) { _ in capturedArtwork }
}
```

**After (correct fix):**
```swift
if let artwork = artwork {
    nowPlayingInfo[MPMediaItemPropertyArtwork] = MPMediaItemArtwork(boundsSize: artwork.size) { @Sendable [artwork] _ in artwork }
}
```

The `@Sendable` attribute explicitly marks the closure as non-isolated, allowing MediaPlayer to call it from any queue.

## Additional Notes
- UIImage is thread-safe for read access after creation, so capturing it in a `@Sendable` closure is safe
- The `@Sendable` attribute is the idiomatic Swift 6 way to mark non-isolated closures
- This is a common Swift 6 migration issue with `MPMediaItemArtwork`
